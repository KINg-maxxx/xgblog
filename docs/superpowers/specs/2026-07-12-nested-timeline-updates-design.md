# 历程子更新设计

## 目标

让每个大历程可以记录多条子更新，同时保持首页时间轴紧凑、清晰、有层次。默认只展示最近两条子更新，读者需要时再展开其余内容；历程及子更新均由现有本地编辑后台维护。

## 数据结构

继续使用 `content/timeline.json` 作为唯一数据源。在现有历程对象上增加可选的 `updates` 数组：

```json
{
  "period": "2026 · 至今",
  "title": "PACT 随访管理平台",
  "text": "项目概述",
  "status": "doing",
  "updates": [
    {
      "date": "2026-07",
      "title": "随访流程更新",
      "text": "完成患者随访流程的调整。"
    }
  ]
}
```

`updates` 按数组顺序展示，最新记录放在最前。`title` 必填，`date` 和 `text` 可为空。没有 `updates` 的旧记录保持原有展示，不需要迁移。

## 前台展示

- 大历程继续使用现有主时间轴、主圆点、阶段、状态、标题和概述。
- 子更新位于概述下方，通过更细、更浅的竖线和小圆点形成主节点的视觉分支。
- 子更新不放入卡片，不增加背景色块，避免层层嵌套。
- 子更新标题使用约 14px 的紧凑字号，日期和描述使用约 13px 的弱化字号。
- 默认展示数组中的前两条。超过两条时显示“展开其余 N 条”；展开后显示全部，并可“收起”。
- 展开按钮带方向箭头、明确的 `aria-expanded` 状态和关联区域；键盘可以正常操作。
- 展开与收起使用轻微的透明度和位移动效，并遵守系统的减少动态效果设置。
- 移动端缩短左侧缩进和间距，文字自然换行，不出现横向滚动。

## 编辑后台

- 每个大历程编辑区内增加“子更新”区域和“添加子更新”按钮。
- 每条子更新可编辑日期、标题和描述，并可上移、下移或删除。
- 新增子更新插入列表顶部，符合“最新在前”的排序习惯。
- 保存时由现有 `/api/timeline` 接口一并校验和写入 `updates`。
- 后台只扩展现有历程编辑器，不新增页面、依赖或第二套存储。

## 兼容与错误处理

- 读取旧数据时，将缺失或非法的 `updates` 视为空数组。
- 保存时清理单行字段并拒绝缺少标题的子更新，错误信息指出对应的大历程和子更新序号。
- 子更新为空时不渲染分支、折叠按钮或额外留白。

## 验证

- 服务端测试覆盖子更新的保存、字段清理、缺少标题和旧数据兼容。
- 首页烟雾测试覆盖子更新结构、默认两条和折叠控制。
- 完整运行现有测试与生产构建。
- 在桌面和移动端检查主次层级、折叠交互、换行、焦点状态和减少动态效果。

## 范围限制

本次不增加富文本、附件、独立详情页、自动日期排序或多层嵌套。子更新只支持一层，顺序完全由编辑后台控制。
